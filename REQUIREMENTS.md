# 白泽（Baize）需求文档

## 1. 项目定位

白泽是**智能体治理基础设施** — Agent 的物流中心与追溯体系。

- **白泽（Baize）**：泽图协议的开源实现，提供智能体注册、数据追溯、行为管控等治理能力
- **泽图协议（Zétú Protocol）**：最简可扩展协议，4 种对象（blob、commit、ref、label），定义智能体交互数据的存储、版本化、查询标准
- **定位对标**：食品/药品追溯码体系 — 信息流即物流，每一步可追溯、可验证、可召回

### 1.1 与政策的关系

白泽是以下政策要求的开源技术底座实现：

| 政策要求 | 白泽实现 |
|---------|---------|
| 智能体数字身份管理（第4条） | X.509 证书链，Agent 注册体系 |
| 能力声明（第4条） | Scope 声明与检查 |
| 决策权限（第6条） | 三层决策模型（自主/授权/用户） |
| 行为管控与围栏（第7条） | Scope 边界 + Zone 数据保护 |
| 可验证可追溯（第7条） | 追溯码体系（blob hash + 证书签名） |
| 安全评估（第8条） | 全链路审计 |
| 供应链安全（第9条） | 沙箱隔离，外部数据检疫 |
| 分类分级治理（第11条） | Level 分级 + Zone 分区 |

## 2. 核心架构

白泽 = 物流中心。数据是货物。信息流即物流。

### 2.1 参与者

| 参与者 | 认证方式 | 隔离 |
|--------|---------|------|
| 用户（User） | 进程身份 / 本地信任 | 无（最高权限） |
| Agent | X.509 证书 + Agent ID | 按 scope 限制 |
| 外部系统 | API Key / 签名 | 沙箱隔离 |

### 2.2 五个关口

| 关口 | 作用 | 对应泽图操作 |
|------|------|------------|
| 收货台 | 入站验证 + 入库 | blob/write |
| 发货台 | 出站验证 + 出库 | blob/query |
| 仓库 | 存储 + 库存管理 | blob/read, commit, ref |
| 隔离区 | 外部数据检疫 | blob/write（外部来源） |
| 调度中心 | Agent 注册与授权 | 证书签发 + scope 管理 |

每个关口执行标准流程：**验身份 → 查权限 → 执行 → 留痕**。

### 2.3 追溯体系

映射食品/药品追溯码，三链并行、可交叉验证：

| 链 | 内容 | 作用 |
|----|------|------|
| 数据链 | blob hash + parent label | 数据从哪来、到哪去 |
| 身份链 | X.509 证书链 | 谁操作的、谁授权的 |
| 治理链 | 审计 blob | 哪些审批、哪些决策 |

| 追溯系统 | 白泽实现 |
|---------|---------|
| 追溯码 | blob hash（SHA-256，唯一不可伪造） |
| 交接记录 | blob + label（from/to/parent） |
| 批次管理 | commit（一组相关 blob 的快照） |
| 经手人签名 | 证书链（X.509） |
| 质检报告 | 评估 blob |
| 召回机制 | 证书撤销 |
| 正向/反向追溯 | parent ↔ children |

## 3. 协议层

### 3.1 泽图 v0（不可变底座）

4 种对象和基本操作：

| 对象 | 用途 |
|------|------|
| blob | 内容寻址的不可变数据单元（SHA-256） |
| commit | 一组 blob 的快照 + 元数据 |
| ref | 指向 commit 的命名引用（分支/标签） |
| label | 挂在 blob/commit 上的 append-only key-value 元数据 |

Push/Pull 是元操作：
- **push = blob/write** — Agent 写入交互数据
- **pull = blob/query** — Agent 按条件查询数据

不需要额外的同步协议，v0 已完整。

### 3.2 泽图 v0e1（治理扩展）

治理能力通过 v0 的 label 扩展机制和传输层证书实现，**不修改 v0 协议本身**：

| 治理能力 | 实现方式 |
|---------|---------|
| 交接签名 | label: `x-signature` |
| 召回机制 | label: `x-status: revoked` |
| Scope 声明 | 证书字段 + label: `x-scope-level`, `x-scope-zones` |
| 身份追踪 | label: `x-agent-id`, `x-parent-agent` |
| 审计留痕 | blob 不可变 + label 追溯链 |
| 身份验证 | 传输层 X.509，不在协议内 |

## 4. 身份体系

### 4.1 Agent 注册

- 所有 Agent 必须注册，包括子 Agent
- Agent 有层级关系：用户 → Agent → Sub-Agent → ...
- 子 Agent 由父 Agent 注册，白泽记录父子关系
- 注册方式双轨：框架支持 hook 则自动注册，不支持则显式注册

### 4.2 证书体系

- X.509 证书链，标准签名算法
- 白泽作为 Root CA，签发顶层 Agent 证书
- Agent 签发子 Agent 证书，形成逐级签发的证书链
- 请求验证时沿证书链追溯到 Root

```
白泽（Root CA）
  └── 签发 → Agent A 证书
        └── 签发 → Sub-Agent A1 证书
        └── 签发 → Sub-Agent A2 证书
  └── 签发 → Agent B 证书
```

### 4.3 证书有效期

双层架构：

- **白泽（体系外粒度）**：定义顶层 Agent 的证书边界（Session / 长期运行等粗粒度）
- **父 Agent（自适应粒度）**：根据子 Agent 的任务性质自行决定粒度

两种模式：
- **时间期限** — 固定时间段（长期运行的服务型 Agent）
- **任务期限** — 绑定任务/Session/Turn（任务型 Agent）

## 5. 权限模型

### 5.1 两种权限

- **分级安全权限（纵向 Level）**：安全等级，决定可用区域的上限
- **特定区域权限（横向 Zone）**：Level 内的子块，决定具体能操作什么

### 5.2 安全等级

| Level | 含义 | 可用区域上限 |
|-------|------|------------|
| 4 | 用户级（最高，信任源头） | 全部区域 |
| 3 | 核心操作（系统级读写、证书签发） | 大部分区域 |
| 2 | 标准操作（常规数据读写、任务执行） | 标准区域 |
| 1 | 受限操作（只读、受限写） | 最小区域 |
| 0 | 隔离区（外部数据，需检疫后升级） | 沙箱区 |

### 5.3 区域定义

区域是 Level 内部的子块。Level 决定了可用区域的上限范围：

```
Level 4 → Zone A, B, C, D, E, ...
Level 3 → Zone A, B, C, D
Level 2 → Zone A, B
Level 1 → Zone A
Level 0 → 沙箱区
```

具体区域由用户/需求定义，白泽只提供 Zone 的注册、分配、检查机制。

Zone 内的数据有保护等级：
- **只读**：系统配置、关键文件，Agent 可读不可改
- **可写**：工作区域，Agent 可读写
- **备份必改**：Agent 可修改，修改前自动备份原文件

不安全的数据不开放给 Agent。在 scope 内的重要数据应有备份或只读保护。

### 5.4 Scope 递减规则

子 Agent 的 scope 永远不超过父 Agent：

- **Level 只能降**：父 Level 3 → 子 Level ≤ 3
- **Zone 只能收窄**：子 Zone 必须 ⊆ 新 Level 允许的范围，且 ⊆ 父 Agent 实际持有的 Zone

```
Agent A:  Level 3, zones: [A, B, C, D]
  ├── Sub-Agent A1: Level 2, zones: [A]       ← Level 降了，Zone 缩了
  └── Sub-Agent A2: Level 2, zones: [A, B]    ← Level 降了，Zone 用满 Level 2 上限
```

### 5.5 临时借权（Scope Elevation）

Agent 可以临时申请扩展 scope。

**借权访问模式：**

| 模式 | 风险 | 审批要求 |
|------|------|---------|
| 只读 | 低 | **默认授予** |
| 只写 | 中 | 需明确理由 |
| 读写 | 高 | 严格审批，需充分理由 |

**借权流程：**

```
① 申请 — 声明需要的 Zone、访问模式、原因、期限
② 审批 — 按层级上报
   请求 ⊆ 父 Agent scope → 父 Agent 审批
   请求超出父 Agent scope → 上报白泽
   请求超出白泽范围    → 上报用户
③ 授予 — 默认只读，写入需额外审批
④ 使用 — Agent 在扩展 scope 内操作
⑤ 归还 — 自动回收 scope + 扫描清理 workspace 中超出权限的数据
```

**路径统一：**

路径 A（临时借权）是基础机制。路径 B（协调 Agent 代管）是路径 A 的特例 — 协调 Agent 用自己已有的权限代为注册，本质上行使的是借权审批权。

## 6. 决策模型

三层决策从 scope + elevation 机制中自然涌现，不需要额外系统：

```
Agent 发起操作
    ├── 在 scope 内           → 自主决策（直接执行，事后审计）
    ├── 超出 scope，可借权     → 授权决策（scope elevation，事前审批）
    └── 超出 scope，不可借权   → 用户决策（上报用户确认）
```

具体哪些操作属于哪一层，由 scope 配置决定，不由白泽预定义。

### 6.1 自主决策

Agent 在 scope 内自行判断、自行行动，不需要事前确认。

保障机制：
- **行为围栏** — scope 限制操作范围，控制爆炸半径
- **事后审计** — 每个操作不可变记录，出事可追溯
- **证书撤销** — 发现问题可立即吊销，止损
- **数据保护** — Zone 内只读/可写/备份必改分级

责任链：Agent 执行责任 → scope 分配者的授权责任 → 用户的最终责任（与证书链一致）。

### 6.2 授权决策

Agent 在**被明确授予的范围**内自行行动。权力来源是派生的，不是固有的。

与自主决策的区别：
- 自主决策：注册时分配的 scope → 内在权力
- 授权决策：被额外授予的 scope → 派生权力

每次授权必须产生治理 blob，记录：

```
who:    谁授权的（签发者证书）
to:     授权给谁（被授权 Agent）
what:   授权了什么（scope 变更）
why:    为什么授权（原因/任务描述）
when:   授权时间
until:  授权期限
mode:   访问模式（只读/只写/读写）
```

授权边界四维度：空间（Zone）、时间（期限）、操作（读写模式）、数据（blob/label 范围）。

授权撤销触发：期限到达、任务完成、授权者主动撤销、异常检测。撤销后必须清理 workspace。

### 6.3 用户决策

没有 Agent 有权做的决策，必须用户亲自拍板。**不可代理、不可绕过、必须留痕。**

两个方向：

**初始定义（自上而下）** — 用户建立治理框架：
- 注册 Agent、分配 scope
- 定义 Zone 含义
- 配置策略（什么操作需要哪层决策）
- 设置证书根
- 建立整体治理规则

**权限兜底（自下而上）** — Agent 链条无法裁决时上报用户：
- 超出所有 Agent scope
- 高风险操作（外部导出、破坏性操作）
- Agent 争议/异常上报
- 策略未覆盖的新情况

用户决策流程：

```
① 上报 — 沿证书链逐级上报到白泽
② 呈现 — 白泽将请求呈现给用户（谁、想做什么、为什么、风险）
③ 用户判断 — 批准 / 拒绝 / 修改
④ 记录 — 用户决策产生治理 blob
```

用户决策的两种形态：
- **批准 scope**：用户授权 scope 变更，之后 Agent 在新 scope 内自主
- **批准操作**：用户直接审批具体操作，一次性

用户不可用时：操作阻塞等待，任何 Agent 不能代替用户决策。可配置超时策略。

**治理闭环：**

```
① 初始定义 — 用户建立规则
② 规则内运行 — Agent 自主/授权决策
③ 规则不够 — 权限兜底，用户裁决
④ 规则更新 — 用户根据经验调整定义 → 回到①
```

## 7. 数据管理

### 7.1 数据边界

白泽只存储**有治理意义的数据**，不存储 Agent 内部过程数据。

| 存储 | 不存储 |
|------|--------|
| 发给 Agent 的 prompt | Agent 内部的 tool call |
| Agent 的最终结果 | Agent 内部的中间推理 |
| 交接元数据（何时、从谁、到谁、scope） | Agent 内部的 prompt-response 循环 |
| 评估/决策记录 | Agent 内部的重试/调试过程 |
| 审计记录 | |

**物流中心只管货物进出，不管工厂怎么生产。**

存储量 = 交接次数 × 交接记录大小，不随 Agent 内部复杂度膨胀。

blob 本身即为高质量训练数据（content + labels + 追溯链），可直接用于 SFT/RLHF/DPO，无需额外设计。

### 7.2 闭环数据管理

所有外部数据必须先导入白泽，处理后再统一导出。Agent 永远不直接接触外部数据。

```
开环（禁止）: Agent ←直接读写→ 外部系统 → 不可追踪、不可审计、不可控制
闭环（必须）: 外部数据 → 导入白泽 → Agent 在白泽内操作 → 导出白泽 → 外部
```

**导入流程（Inbound）：**

```
外部数据 → 检测（安全、格式、大小）→ 认证（来源验证）→ 标注（metadata 更新）
    ├── 通过 → 进入白泽主仓库
    └── 不通过 → 拒绝或进入沙箱区

非可信数据不可进入白泽仓库。
```

**导出流程（Outbound）：**

```
审批通过的结果 → blob → 导出到外部
记录：谁导出了什么、到哪里、何时
```

### 7.3 Agent 工作目录

每个 Agent 拥有独立的临时工作目录，类似工厂工位。

```
白泽主仓库（永久，不可变）  ← 已提交的 blob
Agent 工作目录（临时）      ← 中间结果、草稿、未提交数据
```

**生命周期：**

```
Agent 注册 → 创建工作目录
Agent 执行 → 在工作目录中操作
Agent 完成 → 最终结果提交到主仓库
Agent 撤销 → 工作目录销毁，主仓库 blob 保留
```

**Workspace 数据安全：**

借权归还时必须清理超出当前 scope 的数据：

```
借权归还流程:
  ① 回收 scope（临时 Zone 移除）
  ② 提交最终结果到主仓库（scope 内的数据保留）
  ③ 扫描 workspace（找出超出当前 scope 的数据）
  ④ 清理（删除不符合当前权限的文件）
  ⑤ 审计（记录清理了什么）
```

借权拿来的数据在上下文中使用是合理的，但不能残留在 workspace 中。

### 7.4 数据操作结论

| 问题 | 结论 |
|------|------|
| 跨 Agent 可见性 | Agent 只能读取主仓库中自身 scope 内的数据 |
| 结果聚合 | 无 Leader 不需要；有 Leader 则各 Agent 行为已分别记录 |
| 数据依赖 | Agent 自行管理，白泽不干预 |
| 结果修正 | v1 和 v2 都记录，保留完整历史 |
| 部分失败 | 失败 Agent 工作目录销毁，其他 Agent 结果保留 |
| 已撤销 Agent 数据 | blob 保留（不可变），工作目录销毁 |
| 数据归属 | 工作目录属于 Agent，提交到主仓库后属于系统 |
| 跨树协作 | 不支持矩阵式管理，通过请求转发或新建 Agent 解决 |

## 8. 组织形态

矩阵式管理不适用于 Agent 系统 — 创建新 Agent 几乎零成本，共享 Agent 反而增加 Token 消耗和错误概率，职责单一的 Agent 更可靠。

跨树协作的标准解法：

```
Agent B 需要 Agent A 的能力
  ├── 路径 1: B → A → A 创建子 Agent 完成（请求转发）
  └── 路径 2: B → 白泽 → 新建独立 Agent（干净 scope）
```

支持的组织形态：

| 形态 | 实现方式 |
|------|---------|
| 层级制 | 证书树 + scope 递减 |
| 扁平制 | 白泽直接管理所有 Agent |
| 项目制 | 任务期限证书 + 工作目录 |
| 专业分工 | Zone 划分 |
| 外包/顾问 | 沙箱 + 受限 scope |
| 临时借调 | Scope Elevation |
